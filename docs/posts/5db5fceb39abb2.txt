<hr>
<p><strong>ส่วนที่ 1: อะไรที่ส่งผลกระทบต่อประสิทธิภาพของฐานข้อมูล (What Impacts Database Performance?)</strong></p>
<p>ส่วนนี้จะอธิบายปัจจัยหลักๆ ที่ส่งผลต่อความเร็วและการตอบสนองของฐานข้อมูล</p>
<ol>
<li>
<p><strong>ตัวชี้วัดหลัก (Key Metrics):</strong> ตัวชี้วัดที่ใช้ประเมินประสิทธิภาพ</p>
<ul>
<li><strong>Query Execution Time:</strong> ระยะเวลาที่ใช้ในการประมวลผลคำสั่ง (query) หนึ่งๆ จนเสร็จสิ้น ยิ่งน้อยยิ่งดี</li>
<li><strong>Throughput:</strong> ปริมาณงานหรือจำนวนคำสั่ง (queries) ที่ฐานข้อมูลสามารถประมวลผลได้สำเร็จในหน่วยเวลาหนึ่งๆ (เช่น จำนวน query ต่อวินาที) ยิ่งมากยิ่งดี</li>
<li><strong>Latency:</strong> ความล่าช้าหรือระยะเวลารอคอยตั้งแต่ส่งคำสั่ง (query) ไปจนถึงเริ่มได้รับการตอบสนอง ยิ่งน้อยยิ่งดี</li>
<li><strong>Resource Utilization:</strong> การใช้ทรัพยากรของระบบ เช่น CPU, หน่วยความจำ (RAM), Disk I/O, Network Bandwidth การใช้งานที่สูงเกินไปอาจบ่งชี้ถึงปัญหาคอขวด</li>
</ul>
</li>
<li>
<p><strong>ประเภทของภาระงาน (Workload Type):</strong> ลักษณะการใช้งานฐานข้อมูลที่แตกต่างกันส่งผลต่อประสิทธิภาพต่างกัน</p>
<ul>
<li><strong>Write-Heavy (เน้นการเขียน):</strong> มีการเพิ่ม (Insert) หรืออัปเดต (Update) ข้อมูลจำนวนมาก มักทำให้เกิดความล่าช้า (latency) สูงขึ้น, เกิดการแย่งชิงทรัพยากร (lock contention), และต้องบำรุงรักษา Index บ่อยครั้ง</li>
<li><strong>Read-Heavy (เน้นการอ่าน):</strong> มีการดึงข้อมูล (Select) จำนวนมาก อาจทำให้เกิดความล่าช้าสูงสำหรับ query ที่ซับซ้อน และเกิด Cache Misses (ข้อมูลที่ต้องการไม่อยู่ในแคช ทำให้ต้องไปอ่านจากดิสก์ซึ่งช้ากว่า)</li>
<li><strong>Delete-Heavy (เน้นการลบ):</strong> มีการลบข้อมูลจำนวนมาก อาจทำให้เกิด Fragmentation (พื้นที่ว่างกระจัดกระจายในไฟล์ข้อมูล ทำให้การเข้าถึงข้อมูลช้าลง) และประสิทธิภาพโดยรวมลดลง</li>
<li><strong>Competing Workload (ภาระงานที่แข่งขันกัน):</strong> การมีภาระงานหลายประเภททำงานพร้อมกัน (เช่น การประมวลผลแบบ real-time พร้อมกับ batch processing) อาจทำให้เกิดการแย่งชิงทรัพยากรและเพิ่มความล่าช้า</li>
</ul>
</li>
<li>
<p><strong>ปัจจัยสำคัญ (Key Factors):</strong> ปัจจัยอื่นๆ ที่ส่งผลต่อการออกแบบและประสิทธิภาพ</p>
<ul>
<li><strong>Item Size:</strong> ขนาดของข้อมูลแต่ละรายการ</li>
<li><strong>Item Type:</strong> ประเภทของข้อมูล (เช่น ข้อความ, ตัวเลข, รูปภาพ)</li>
<li><strong>Dataset Size:</strong> ขนาดโดยรวมของฐานข้อมูล</li>
<li><strong>Concurrency:</strong> จำนวนผู้ใช้หรือโปรเซสที่เข้าถึงฐานข้อมูลพร้อมกัน</li>
<li><strong>Consistency Expectations:</strong> ระดับความถูกต้องและทันสมัยของข้อมูลที่ต้องการ (เช่น ต้องการข้อมูลล่าสุดเสมอ หรือยอมรับข้อมูลที่อาจจะเก่าเล็กน้อยได้)</li>
<li><strong>Geographic Distribution:</strong> การกระจายตัวของผู้ใช้หรือเซิร์ฟเวอร์ในพื้นที่ทางภูมิศาสตร์ต่างๆ</li>
<li><strong>HA Expectations (High Availability):</strong> ความคาดหวังเรื่องความพร้อมใช้งานของระบบ (ต้องการให้ระบบทำงานต่อเนื่องแค่ไหน)</li>
<li><strong>Workload Variability:</strong> ความผันผวนของภาระงานในช่วงเวลาต่างๆ</li>
</ul>
</li>
</ol>
<hr>
<p><strong>ส่วนที่ 2: กลยุทธ์เพื่อเพิ่มประสิทธิภาพฐานข้อมูล (Database Performance Strategies)</strong></p>
<p>ส่วนนี้จะแนะนำเทคนิคและวิธีการต่างๆ ที่ใช้ในการปรับปรุงประสิทธิภาพฐานข้อมูล</p>
<ol>
<li>
<p><strong>Database Indexing (การทำดัชนี):</strong></p>
<ul>
<li><strong>หลักการ:</strong> สร้างโครงสร้างข้อมูลพิเศษ (เรียกว่า Index) ที่ช่วยให้ค้นหาข้อมูลในตารางได้เร็วขึ้น เหมือนสารบัญในหนังสือ</li>
<li><strong>ตัวอย่าง:</strong> สร้าง Index บนคอลัมน์ 'Email' โดย Index จะเก็บค่า Email และตัวชี้ (Pointer) ไปยังแถวข้อมูลจริง ทำให้เมื่อค้นหาด้วย Email จะหาเจอได้เร็วโดยไม่ต้องสแกนทั้งตาราง</li>
<li><strong>ประโยชน์:</strong> ลดเวลา Query Execution Time สำหรับการค้นหาและเงื่อนไข WHERE</li>
</ul>
</li>
<li>
<p><strong>Sharding and Partitioning (การแบ่ง Shard และ Partition):</strong></p>
<ul>
<li><strong>หลักการ:</strong> แบ่งฐานข้อมูลขนาดใหญ่ออกเป็นส่วนเล็กๆ ที่เรียกว่า Shard หรือ Partition ซึ่งจัดการได้ง่ายและเร็วกว่า Shard มักจะกระจายอยู่บนหลายๆ เซิร์ฟเวอร์ ส่วน Partition มักจะอยู่บนเซิร์ฟเวอร์เดียวกันแต่แบ่งตามตรรกะ</li>
<li><strong>ตัวอย่าง:</strong> แบ่งฐานข้อมูล Monolithic (ก้อนเดียว) ออกเป็นหลายๆ Shard ทำให้ข้อมูลกระจายตัวออกไป</li>
<li><strong>ประโยชน์:</strong> เพิ่มความสามารถในการขยายระบบ (Scalability), เพิ่ม Throughput, ลดภาระงานในแต่ละส่วน</li>
</ul>
</li>
<li>
<p><strong>Denormalization (การลดทอน Normalization):</strong></p>
<ul>
<li><strong>หลักการ:</strong> เพิ่มข้อมูลที่ซ้ำซ้อนเข้าไปในตาราง เพื่อหลีกเลี่ยงการ Join ตารางหลายๆ ตารางเข้าด้วยกัน ซึ่งมักจะช้า</li>
<li><strong>ตัวอย่าง:</strong> แทนที่จะเก็บข้อมูล Products, Segments, Customers, Orders แยกกัน แล้วต้อง Join เพื่อดูข้อมูลลูกค้าและการสั่งซื้อ ก็สร้างตาราง <code>Customer_Orders</code> ใหม่ที่รวมข้อมูลสำคัญจากตารางเหล่านั้นมาไว้ด้วยกันเลย</li>
<li><strong>ประโยชน์:</strong> เพิ่มความเร็วในการอ่านข้อมูล (Read-Heavy workload) เพราะลดความจำเป็นในการ Join</li>
<li><strong>ข้อเสีย:</strong> เพิ่มความซับซ้อนในการเขียนข้อมูล (ต้องอัปเดตข้อมูลซ้ำซ้อนในหลายที่) และเปลืองพื้นที่เก็บข้อมูล</li>
</ul>
</li>
<li>
<p><strong>Database Replication (การจำลองฐานข้อมูล):</strong></p>
<ul>
<li><strong>หลักการ:</strong> สร้างสำเนาของฐานข้อมูล (Replica) ขึ้นมาหลายๆ ชุด</li>
<li><strong>โครงสร้าง:</strong>
<ul>
<li><strong>Leader Node (หรือ Master):</strong> โหนดหลัก รับผิดชอบการเขียน (Write) และอัปเดตข้อมูล</li>
<li><strong>Follower Nodes (หรือ Slaves/Replicas):</strong> โหนดสำเนา รับข้อมูลที่เปลี่ยนแปลงจาก Leader ผ่าน Replication Stream และมักใช้สำหรับรองรับการอ่าน (Read-Only)</li>
</ul>
</li>
<li><strong>กระบวนการ:</strong> ข้อมูลที่เขียนลง Leader จะถูกส่ง (Replicate) ไปยัง Follower Nodes</li>
<li><strong>ประโยชน์:</strong> กระจายภาระการอ่าน (Load Balancing), เพิ่มความพร้อมใช้งาน (High Availability - ถ้า Leader ล่ม Follower สามารถทำงานแทนได้), ลด Latency สำหรับผู้ใช้ที่อยู่ใกล้ Follower Node</li>
</ul>
</li>
<li>
<p><strong>Database Locking Techniques (เทคนิคการล็อก):</strong></p>
<ul>
<li><strong>หลักการ:</strong> กลไกในการจัดการการเข้าถึงข้อมูลพร้อมกัน (Concurrency) เพื่อป้องกันปัญหาข้อมูลผิดพลาด (เช่น Lost Update, Dirty Read)</li>
<li><strong>ตัวอย่าง (Optimistic Locking):</strong> เป็นการล็อกแบบคาดการณ์ในแง่ดีว่าไม่น่าจะเกิด conflict บ่อย
<ol>
<li><strong>Sarah อ่านข้อมูล:</strong> ดึงข้อมูลบัญชี ID: 1, จำนวนเงิน: 40, เวอร์ชัน: 1</li>
<li><strong>John อ่านข้อมูล:</strong> ดึงข้อมูลเดียวกัน (ID: 1, Amt: 40, Version: 1)</li>
<li><strong>John อัปเดตข้อมูล:</strong> ลดจำนวนเงิน 20 โดยระบุว่าต้องเป็นเวอร์ชัน 1 --> อัปเดตสำเร็จ บัญชีกลายเป็น Amt: 20, <strong>Version: 2</strong> และทำการ COMMIT (ยืนยันการเปลี่ยนแปลง)</li>
<li><strong>Sarah พยายามอัปเดตข้อมูล:</strong> พยายามลดจำนวนเงิน 20 โดยระบุว่าต้องเป็นเวอร์ชัน 1 --> <strong>อัปเดตไม่สำเร็จ</strong> เพราะข้อมูลในฐานข้อมูลตอนนี้เป็นเวอร์ชัน 2 แล้ว (No Account Row Matched) แสดงว่ามีคนอื่นแก้ไขไปก่อนหน้า</li>
</ol>
</li>
<li><strong>ประโยชน์:</strong> ช่วยรักษาความถูกต้องของข้อมูล (Data Integrity) เมื่อมีผู้ใช้หลายคนเข้าถึงและแก้ไขข้อมูลพร้อมกัน Optimistic Locking ลดการบล็อกกัน แต่ต้องมีกลไกจัดการเมื่อเกิด conflict</li>
</ul>
</li>
</ol>
